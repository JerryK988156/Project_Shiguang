# 拾光计划项目规划

## 1. 项目概述

### 1.1 项目名称
拾光计划

### 1.2 项目定位
拾光计划是一个面向学生和个人学习者的学习打卡与目标管理系统，帮助用户设定学习目标、进行每日打卡、记录学习内容，并通过统计分析了解自己的学习习惯与执行情况。

### 1.3 项目目标
- 支持用户创建和管理学习目标
- 支持按天进行学习打卡并记录学习内容
- 支持用户创建任意数量的学习任务，并对每个任务独立跟踪完成情况
- 支持查看目标进度、累计学习时长、连续打卡天数
- 支持通过图表展示近期学习趋势
- 提供简单后台管理能力，便于展示完整前后端项目结构

### 1.4 适用场景
- 学生日常学习计划管理
- 考研、考证、自学阶段的目标追踪
- 课程作业中的中小型前后端项目展示

## 2. 项目范围

### 2.1 本期实现范围
- 用户登录、退出、获取当前用户信息
- 学习目标的新增、编辑、删除、查看、状态切换
- 每日打卡、查看打卡记录、删除打卡记录
- 个人学习统计与趋势展示
- 管理员查看用户列表与系统概览数据
- 前后端分离部署与联调
- 代码托管到 GitHub

### 2.2 暂不实现
- 短信、邮件、微信提醒
- 多级复杂权限系统
- 真正的消息推送
- 文件云存储
- 社交互动、评论、点赞
- AI 学习建议

## 3. 用户角色

### 3.1 普通用户
- 登录系统
- 管理自己的学习目标
- 每日打卡
- 查看个人统计数据
- 修改个人资料和密码

### 3.2 管理员
- 查看用户列表
- 查看全站目标和打卡概览
- 可选删除异常记录

> 为控制开发周期，角色仅保留 `admin` 和 `user` 两种。

## 4. 技术方案

### 4.1 前端技术栈
- Vue 3
- Vite
- Element Plus
- Vue Router
- Pinia
- Axios
- ECharts
- Sass

### 4.2 后端技术栈
- Spring Boot 3
- MyBatis-Plus
- OceanBase（MySQL 兼容模式）
- Lombok

### 4.3 数据库接入说明
- 本项目后端接入的是 OceanBase 数据库
- 开发时按 MySQL 兼容语法设计表结构与 SQL
- Java 侧仍可使用常见 MySQL 连接方式和驱动进行接入
- 在建表和索引设计时，需要注意兼容 MySQL 语法，避免使用不必要的方言特性

### 4.4 鉴权方案
建议优先采用 Session 登录，原因如下：
- 与当前教学项目风格一致
- 实现成本低，适合 4 到 5 天开发周期
- 足够满足课程项目展示需求

后续如果需要，也可以替换为 JWT。

### 4.5 前后端分离说明
本项目采用前后端分离架构：
- 前端项目负责页面展示、交互逻辑、状态管理和图表渲染
- 后端项目负责登录鉴权、业务处理、数据库访问和统计接口
- 前后端通过 RESTful API 通信，使用 JSON 作为主要数据交换格式
- 前端通过 Axios 调用后端接口，开发阶段可通过 Vite 代理解决跨域问题

## 5. 功能模块规划

## 5.1 用户模块

### 功能点
- 登录
- 退出登录
- 获取当前登录用户信息
- 修改昵称、头像、个人简介
- 修改密码

### 说明
- 初版可不做注册，直接由数据库预置测试账号
- 头像上传可作为加分项，若时间紧可先使用默认头像字段

## 5.2 学习目标模块

### 功能点
- 新建目标
- 编辑目标
- 删除目标
- 查看目标详情
- 查看目标列表
- 设置目标状态

### 目标字段建议
- 目标标题
- 目标说明
- 开始日期
- 结束日期
- 预计坚持天数
- 当前状态

### 状态设计
- 进行中
- 已完成
- 已放弃

## 5.3 打卡模块

### 功能点
- 今日打卡
- 查看打卡记录列表
- 查看是否已完成今日打卡
- 删除打卡记录

### 打卡内容建议
- 关联目标
- 打卡日期
- 学习时长
- 学习内容
- 学习心得或备注

### 业务规则
- 用户可以创建任意数量的学习任务
- 对于同一个任务，用户每天只能打卡一次
- 同一天可以对不同任务分别打卡
- 后端需校验 `(user_id, goal_id, checkin_date)` 的唯一性
- 数据库层建议增加唯一索引，避免重复打卡数据写入

## 5.4 统计分析模块

### 功能点
- 近 7 天打卡趋势
- 近 30 天打卡趋势
- 累计学习时长
- 连续打卡天数
- 目标完成率
- 各目标学习时长占比

### 图表建议
- 折线图：最近 7 天打卡次数 / 学习时长
- 柱状图：各目标累计学习时长
- 饼图：目标状态占比
- 数字卡片：累计时长、连续打卡、目标总数

## 5.5 后台管理模块

### 功能点
- 用户列表
- 目标总览
- 打卡总览
- 系统概览统计

### 后台展示建议
- 用户总数
- 目标总数
- 打卡总数
- 本周活跃用户数

- `/dashboard` 首页看板
- `/goal/list` 目标列表页
- `/goal/edit` 目标新增/编辑页
- `/checkin/add` 今日打卡页

- `/admin/user` 用户管理页
- `/admin/stat` 后台统计页

### 6.2 导航建议
- 首页看板
- 我的目标
- 今日打卡
- 打卡记录
- 统计分析
- 个人中心
- 管理后台（仅管理员可见）

### 6.3 前端职责边界
- 登录页、布局页、表单页、列表页、图表页全部由前端实现
- 前端负责表单校验、交互反馈、路由控制和页面权限显示
- 前端不直接处理数据库逻辑，所有业务数据都通过接口获取

### 6.4 后端职责边界
- 提供登录、用户、目标、打卡、统计等 API
- 负责 Session 校验、业务规则判断、数据库读写和统计聚合
- 对“每个任务每天只能打卡一次”进行统一校验

| 字段名 | 类型 | 说明 |

## 8. 接口规划

## 8.1 认证接口
- `POST /api/auth/login` 用户登录
- `POST /api/auth/logout` 退出登录
- `GET /api/auth/userInfo` 获取当前用户信息

## 8.2 用户接口
- `PUT /api/user/profile` 修改个人资料
- `PUT /api/user/password` 修改密码
- `GET /api/admin/user/list` 管理员查看用户列表

## 8.3 目标接口
- `POST /api/goal/add` 新建目标
- `PUT /api/goal/update` 编辑目标
- `DELETE /api/goal/delete/{id}` 删除目标
- `GET /api/goal/detail/{id}` 目标详情
- `GET /api/goal/list` 目标列表
- `PUT /api/goal/status/{id}` 修改目标状态

## 8.4 打卡接口
- `POST /api/checkin/add` 新增打卡
- `GET /api/checkin/list` 打卡记录列表
- `GET /api/checkin/today` 查询今日各任务打卡状态
- `DELETE /api/checkin/delete/{id}` 删除打卡记录

## 8.5 统计接口
- `GET /api/stat/overview` 个人统计概览
- `GET /api/stat/trend7` 近 7 天趋势
- `GET /api/stat/trend30` 近 30 天趋势
- `GET /api/stat/goalProgress` 目标完成情况
- `GET /api/admin/stat/overview` 管理端系统概览

## 16. 风险与控制

### 风险点
- 如果同时做太多加分功能，容易拖慢进度
- 若登录鉴权改为 JWT，接口联调工作量会略增
- 图表数据维度过多，前端渲染和接口设计会变复杂
- 如果不做数据库唯一约束，重复打卡数据容易污染统计结果
- 若 GitHub 提交拖到最后，容易遗漏文档和历史记录

### 控制建议
- 先完成 MVP，再补加分项
- 管理员功能保持轻量
- 数据统计先做核心指标，不追求过多图表
- 在后端校验之外，数据库层同步建立唯一索引
- 开发过程中分阶段提交 Git，并定期推送远程仓库

## 17. 后续实施建议

后续进入实现阶段时，建议按以下顺序推进：
- 先确定前后端目录结构与 Git 仓库
- 再写 OceanBase 数据库 SQL 和后端实体类
- 接着完成登录与基础结构
- 再完成目标模块
- 然后完成打卡模块和唯一性校验
- 最后做统计、管理端页面与 GitHub 收尾

这样可以保证项目始终处于“可运行、可演示”的状态。
